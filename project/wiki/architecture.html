Finance Agent (发仔) · 字节跳动内部项目

架构设计

Finance Agent（发仔）采用严格的分层架构： API 层 → Agent 层 → 基础设施层。 每一层只能向下依赖，模块间依赖方向严格单向，确保各层职责清晰、可独立测试与替换。

  • API 层（app/api/）：负责 HTTP 路由、认证鉴权与流式响应，是系统对外暴露的唯一入口。
  • Agent 层（app/agent/）：包含 Agent 主循环、提示词构建、LLM 调用、工具注册与上下文管理，是系统的核心推理引擎。
  • 基础设施层（app/infra/）：提供 MongoDB、Redis、TOS 对象存储和沙箱执行等底层能力，不依赖任何上层业务逻辑。

各层详细说明

API 层（app/api/）

API 层是系统对外的统一入口，基于 FastAPI 构建，负责接收来自 Web 前端、 Lark Bot 及 CLI 的所有请求，完成认证鉴权后将请求转发给 Agent 层处理。

FastAPI 路由

所有 HTTP 端点均通过 FastAPI 的路由装饰器注册，路由模块按业务划分： 聊天（chat.py）、会话管理（sessions.py）、 文件操作（files.py）、工具查询（tools.py）。 每个路由返回统一的 JSON 响应格式： 

{
  "code": 0,          // 0 = 成功，非 0 = 错误
  "message": "ok",    // 错误时携带详细说明
  "data": { ... }     // 业务数据载体
}

JWT 认证中间件

所有需要鉴权的路由均通过 FastAPI Dependency Injection 注入 verify_token 依赖，从请求头 Authorization: Bearer <token> 中提取并验证 JWT，解析出 user_id 与权限范围，注入到请求上下文中供下游使用。

SSE 流式输出

聊天接口采用 Server-Sent Events（SSE） 实现流式响应。 Agent 层通过 Redis Streams 向频道广播事件， API 层订阅对应频道并通过 FastAPI 的 StreamingResponse 将事件实时推送给客户端，无需长轮询。 

# 流式响应伪码示意
async def stream_chat(session_id: str):
    async def event_generator():
        async for event in redis.xread(session_id):
            yield f"data: {event}\n\n"
    return StreamingResponse(event_generator(), media_type="text/event-stream")

Agent 层（app/agent/）

Agent 层是发仔的核心推理引擎，负责驱动 LLM 推理、工具调用、上下文管理等全部 Agent 逻辑。 该层完全无 HTTP 依赖，可独立测试。

AgentLoop 主循环（loop.py）

AgentLoop 是 Agent 的驱动核心，实现「感知 → 推理 → 行动 → 观察」的循环迭代：

  1. 接收用户消息，写入会话历史
  2. 调用 PromptBuilder 构建当前轮次的完整 Prompt
  3. 调用 LLM Provider 获取模型响应（支持流式输出）
  4. 解析响应中的工具调用（Tool Call）指令
  5. 通过 ToolRegistry 执行对应工具，将结果追加到消息历史
  6. 若模型返回最终回答（无工具调用）则终止循环，否则继续下一轮
async def run(self, user_message: str) -> AsyncIterator[str]:
    self.session.append_user(user_message)
    while True:
        response = await self.llm.chat(self.prompt_builder.build(self.session))
        if response.has_tool_calls:
            results = await self.registry.execute_all(response.tool_calls)
            self.session.append_tool_results(results)
        else:
            yield response.content
            break

PromptBuilder 系统提示构建（prompt.py）

PromptBuilder 负责在每轮推理前组装完整的消息列表，包括： 系统提示（角色设定、数据权限、当前日期）、历史对话消息、 可用工具描述（Tool Schema）以及当前用户消息。 支持按用户权限动态注入不同数据源的访问凭证与 Schema 信息。

LLM Provider 抽象层

通过统一的 BaseLLMProvider 抽象类屏蔽底层模型差异， 目前支持两个后端：

  • OpenAI：通过官方 Python SDK 调用 GPT-4o 系列模型，支持 Function Calling 与流式输出。
  • VolcEngine Ark：字节跳动内部大模型服务，兼容 OpenAI 接口协议，通过 ArkProvider 适配器接入。

切换模型后端只需修改配置项 LLM_PROVIDER，无需改动 Agent 层其余代码。

ToolRegistry 工具注册与执行（registry.py）

ToolRegistry 维护所有可用工具的注册表，提供：

  • 工具注册：通过 @registry.register 装饰器声明工具名称、参数 Schema 和处理函数。
  • Schema 导出：将注册表中的工具描述格式化为 OpenAI Function Calling 规范的 JSON Schema，注入 Prompt。
  • 并发执行：对模型返回的多个工具调用，使用 asyncio.gather 并发执行以降低延迟。
  • 错误隔离：单个工具执行失败不中断整体流程，错误信息以 tool result 的形式返回给模型。

SessionManager 会话状态管理（session.py）

SessionManager 负责会话的生命周期管理：

  • 从 MongoDB 加载历史会话消息，初始化内存中的消息列表
  • 追加用户消息、助手消息、工具调用结果到消息列表
  • 利用 Redis 分布式锁防止同一会话被并发写入，保证消息顺序一致
  • 定期将内存消息持久化回 MongoDB

TokenTracker 上下文窗口管理（token.py）

TokenTracker 实时追踪当前消息列表的 Token 用量， 在每次调用 LLM 前检查是否接近上下文窗口上限（默认阈值：最大窗口的 85%）。 当 Token 使用量超出阈值时，自动触发 ContextCompaction 进行消息压缩。

Context Compaction 消息历史压缩（compaction.py）

当消息历史过长时，ContextCompaction 会对历史消息执行摘要压缩：

  1. 保留系统提示与最近 N 轮对话（默认保留最近 10 轮）
  2. 对较早的历史消息调用 LLM 生成压缩摘要（Summary Message）
  3. 用摘要消息替换原始历史，大幅缩减 Token 占用
  4. 压缩后的摘要以特殊角色 system（压缩摘要）写回会话

基础设施层（app/infra/）

基础设施层封装所有外部存储与执行依赖，向 Agent 层暴露简洁的异步接口， 屏蔽底层实现细节。该层不依赖任何业务逻辑，可单独进行集成测试。

MongoDB（ByteDoc）- 消息 / 会话 / 文件元数据

MongoDB 是系统的主持久化存储，使用字节跳动内部的 ByteDoc 托管服务。主要存储：

  • sessions 集合：会话元数据（创建时间、用户 ID、标题、状态）
  • messages 集合：会话消息历史（角色、内容、工具调用、时间戳）
  • files 集合：用户上传文件及 Agent 产物的元数据（文件名、TOS 路径、MIME 类型、大小）

Redis - 会话锁 + SSE Stream + 热缓存

Redis 在系统中承担三类职责：

  • 分布式会话锁：使用 SET NX PX 原语实现悲观锁，确保同一会话的并发写操作串行化，防止消息乱序。
  • SSE Streams：Agent 层将流式 token、工具调用事件写入 Redis Stream（XADD），API 层消费（XREAD）并推送给客户端，实现跨进程的事件广播。
  • 热缓存：频繁访问的会话消息列表、用户权限信息以短 TTL（60 s）缓存于 Redis，减少 MongoDB 查询压力。

TOS 对象存储 - 用户上传文件 + Agent 产物

TOS（字节跳动对象存储）存储所有二进制文件内容：

  • 用户上传的 Excel、CSV、PDF 等原始文件
  • Agent 生成的图表（PNG/SVG）、分析报告（PDF/DOCX）等产物
  • 文件按 workspace/{user_id}/{session_id}/{file_id} 路径组织，确保多用户隔离
  • 下载链接通过签名 URL（Pre-signed URL）生成，有效期默认 1 小时

Sandbox - 沙箱执行环境

沙箱模块封装字节跳动内部 ByteAI Sandbox 服务，提供安全隔离的代码执行环境：

  • 支持 Python（数据分析、可视化）与 Shell 脚本执行
  • 每次执行在独立容器中运行，执行完毕后销毁，无状态残留
  • 网络访问、文件系统写入均受严格限制，防止数据泄露与恶意操作
  • 执行结果（stdout、stderr、生成文件）通过 TOS 回传给 Agent
  • 执行超时默认 60 秒，可按工具级别配置

模块依赖方向

系统严格遵循单向依赖原则：上层模块可以依赖下层， 下层模块绝不反向依赖上层。这一设计使各层可以独立测试， 并允许在不影响上层的情况下替换基础设施实现。 

  app/api/           ←── 仅此层对外暴露 HTTP 接口
      │
      │  depends on
      ▼
  app/agent/         ←── 纯业务逻辑，无 HTTP/网络依赖
      │
      │  depends on
      ▼
  app/infra/         ←── 纯基础设施，无业务逻辑

  ✅ api    → agent  （允许）
  ✅ api    → infra  （允许，直接读取配置等）
  ✅ agent  → infra  （允许）
  ❌ agent  → api    （禁止，反向依赖）
  ❌ infra  → agent  （禁止，反向依赖）
  ❌ infra  → api    （禁止，反向依赖）

实时流式架构（SSE）

发仔的流式响应基于 Redis Streams + FastAPI StreamingResponse + 前端 EventSource 三层管道实现，解耦了 Agent 推理进程与 HTTP 响应进程，支持水平扩展。

完整数据流

  用户发送消息（POST /api/chat）
        │
        ▼
  API 层创建 SSE 响应，返回 StreamingResponse
        │
        ├─── 后台异步启动 AgentLoop（独立协程 / Worker）
        │
        ▼
  AgentLoop 推理过程中，将事件写入 Redis Stream
        │
        │  XADD session:{session_id} * type token data "Hello"
        │  XADD session:{session_id} * type tool_call data {...}
        │  XADD session:{session_id} * type done data ""
        │
        ▼
  API 层 SSE Generator 订阅 Redis Stream（XREAD BLOCK）
        │
        │  持续读取新事件，格式化为 SSE 协议：
        │  data: {"type":"token","data":"Hello"}\n\n
        │
        ▼
  前端 EventSource 接收事件，实时渲染流式输出
        │
        │  source.addEventListener('message', (e) => {
        │    const { type, data } = JSON.parse(e.data);
        │    if (type === 'token') appendToken(data);
        │  });
        │
        ▼
  收到 done 事件后，EventSource 关闭连接

SSE 事件类型

事件类型（type） 触发时机 data 字段
token LLM 流式输出每个 token 时 token 文本片段
tool_call Agent 发起工具调用时 工具名称与参数 JSON
tool_result 工具执行完成后 工具执行结果摘要
error Agent 执行异常时 错误信息字符串
done Agent 本轮推理全部完成 空字符串

设计优势：Agent 推理进程与 HTTP 连接进程完全解耦。 即使客户端断开重连，只需重新订阅同一 Redis Stream（通过记录上次读取的 offset）， 即可从断点继续接收未读事件，实现断点续传。
