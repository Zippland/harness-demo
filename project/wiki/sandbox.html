沙箱环境

发仔的沙箱执行环境为 Agent 的 Shell/Python 代码提供隔离的运行空间， 确保用户数据安全、系统稳定，并防止恶意代码对宿主环境造成影响。 每个会话独占一个沙箱实例，沙箱之间互不可见，彻底杜绝跨会话数据泄露。

沙箱概述

沙箱的核心目标是在不限制 Agent 合理计算能力的前提下，最小化安全攻击面。具体设计原则如下：

  • 代码隔离：Agent 通过 shell_exec 工具触发的所有 Shell 命令与 Python 脚本，均在独立沙箱容器中执行，不直接访问宿主机文件系统或网络。
  • 双 Provider 架构：生产环境使用 ByteAI Sandbox（gVisor 内核级隔离），开发与测试环境使用 Local Docker Sandbox（标准 Docker 容器），两者对上层代码暴露统一接口。
  • 会话级生命周期：沙箱随会话创建而分配，随会话删除（或超时）而回收， 生命周期与用户会话严格绑定。

Provider 切换：通过环境变量 SANDBOX_PROVIDER 指定使用的沙箱实现。 设置为 bytedai 启用生产级 ByteAI Sandbox，设置为 local 启用本地 Docker 沙箱。

沙箱工作目录结构

每个沙箱实例在启动时会初始化一个标准化的 /workspace/ 工作目录， Agent 的所有文件读写操作均限制在此目录下，各子目录职责明确： 

/workspace/
  uploads/      # 用户上传的文件（CSV、Excel、图片等原始数据）
  artifacts/    # Agent 产出的分析结果（图表、报告、生成文件等）
  skills/       # 加载的 Skills 包（动态扩展能力模块）
  sessions/     # 会话相关文件（上下文快照、临时状态等）
  memory.json   # Agent 持久化记忆（跨轮次保存的关键信息）

目录说明：

  • uploads/：用户通过前端上传的文件会被同步至此目录， Agent 可直接读取进行数据分析，该目录对 Agent 只读（防止意外覆盖用户原始数据）。
  • artifacts/：Agent 执行分析后产生的所有输出文件写入此目录， 前端通过 Artifacts API 轮询并展示给用户。
  • skills/：SkillTool 动态加载的技能包解压至此目录， Agent 可调用技能包中预定义的分析逻辑。
  • sessions/：存储会话级的临时状态文件， 如任务进度、中间计算结果等，会话结束后随沙箱一同清理。
  • memory.json：由 MemoryTool 维护， 存储 Agent 在当前会话中主动持久化的关键信息，支持跨轮次记忆检索。

沙箱生命周期

沙箱的完整生命周期分为六个阶段，从会话创建到资源清理形成闭环：

  1. 会话创建 → 分配沙箱
    用户发起新会话时，sandbox/manager.py 中的沙箱池管理器优先从空闲池中取出 一个预热的沙箱实例；若池中无可用实例，则动态创建新沙箱并等待其就绪。
  2. 准备 workspace/ 目录
    沙箱分配完成后，系统自动初始化标准工作目录结构， 创建 uploads/、artifacts/、skills/、 sessions/ 等子目录，并生成空白的 memory.json。
  3. 用户上传文件 → 同步到 /workspace/uploads/
    用户通过前端上传的文件经由后端文件服务同步到对应沙箱的 /workspace/uploads/ 目录，Agent 随即可以访问这些文件。
  4. Agent 执行 shell_exec 工具 → 在沙箱中运行
    Agent 调用 ExecTool（shell_exec）时，命令被路由到该会话绑定的 沙箱实例中执行。执行结果（标准输出、标准错误、退出码）实时返回给 Agent。
  5. 产物输出到 /workspace/artifacts/
    Agent 在沙箱中生成的图表、报告等产物写入 /workspace/artifacts/， 后端定期扫描此目录并通过 Artifacts API 将新文件推送给前端展示。
  6. 会话删除 → TTL 后清理沙箱
    会话结束或超时后，沙箱进入待回收状态。定时任务 cronjob/sandbox_session_reconcile.py 负责扫描过期沙箱， 释放计算资源并清理所有关联的 workspace 数据。

安全机制

沙箱通过多层防御机制保障执行安全，各层机制相互补充，形成纵深防御体系：

  • 🛡️ gVisor（系统调用过滤）：在 ByteAI Sandbox 中， 容器内的系统调用由 gVisor 用户态内核拦截并审查，攻击者无法直接触达宿主机内核， 大幅降低内核漏洞利用风险。
  • 📂 只读根文件系统：容器的根文件系统以只读方式挂载， 恶意代码无法篡改系统二进制文件或注入持久化后门；仅 /workspace/ 挂载为可写 Volume。
  • 🌐 网络限制（默认禁用，可配置白名单）：沙箱默认关闭出站网络访问， 防止数据外泄和 C&C 回连；如业务需要，可通过白名单配置放行特定域名或 IP 段。
  • ⚙️ CPU / 内存资源限制：通过 cgroup 对每个沙箱实例设置 CPU 配额与内存上限， 避免单个 Agent 任务耗尽宿主机资源，影响其他用户的服务质量。
  • 🔢 PID 数量限制：限制沙箱内可创建的进程与线程总数， 防止 fork 炸弹等拒绝服务攻击导致宿主机资源耗尽。
  • 🗂️ 每会话独立隔离 Volume：每个沙箱实例拥有独立的存储 Volume， 不同会话之间的文件系统完全隔离，确保用户数据不会被其他会话访问或污染。

沙箱 Provider 对比

发仔支持两种沙箱 Provider，分别针对生产环境的安全性需求与本地开发的便利性需求进行了优化：

特性 ByteAI Sandbox Local Docker Sandbox
使用场景 生产环境 开发 / 测试
隔离级别 gVisor（内核级） Docker（容器级）
配置方式 SANDBOX_PROVIDER=bytedai SANDBOX_PROVIDER=local
网络策略 受限白名单（默认禁用） 可自由配置
启动速度 沙箱池预热，毫秒级分配 按需创建，秒级启动
依赖要求 ByteAI 平台服务 本地 Docker Engine

相关配置项

沙箱相关的配置项与核心模块如下：

环境变量

配置项 可选值 说明
SANDBOX_PROVIDER bytedai | local 指定沙箱实现。生产环境使用 bytedai，本地开发使用 local

核心模块

  • sandbox/manager.py：沙箱池管理器， 负责沙箱实例的创建、分配、回收与健康检查。维护空闲池以缩短会话启动延迟。
  • cronjob/sandbox_session_reconcile.py： 定时清理任务，定期扫描数据库中已删除或超时（TTL 到期）的会话， 调用沙箱管理器释放对应的沙箱实例与存储 Volume，防止资源泄漏。

本地开发提示：使用 Local Docker Sandbox 时，请确保本地已安装并启动 Docker Engine，且当前用户有权限执行 docker 命令。 可通过 docker ps 验证 Docker 是否正常运行。
