Agent 循环

AgentLoop 是发仔系统的核心，负责协调 LLM 调用与工具执行的主循环。 它将用户输入、会话历史、LLM 推理和工具执行有机串联，驱动整个对话与任务完成流程。

1. Agent Loop 概述

AgentLoop 以事件驱动的方式运行，每一轮循环从接收用户消息开始， 经过会话加载、Prompt 构建、LLM 流式推理、工具执行，再到消息持久化， 直到 LLM 输出纯文本回复（无工具调用）或达到终止条件为止。

核心职责包括：

  • 协调 SessionManager 加载和维护会话上下文
  • 驱动 LLM Provider 进行流式推理，处理 thinking、text、tool_calls 三类事件
  • 调度 ToolRegistry 执行工具调用，并将结果写回上下文
  • 通过 MessageStore 将每一步执行结果实时持久化至 MongoDB
  • 管理 Token 用量，必要时自动压缩历史上下文

2. 执行流程

Agent Loop 的完整执行流程如下所示，每轮循环按编号步骤顺序推进：

  1. 接收用户输入：从 API 层接收用户消息，触发一轮新的 Agent Loop。
  2. 加载会话历史：SessionManager 从 MongoDB 读取历史消息，并利用 Redis 热缓存加速访问，构建完整的对话上下文。
  3. 构建 Prompt：PromptBuilder 将 system prompt（包含角色设定、工具列表、当前日期等元信息）与消息历史合并，组装为 LLM 输入。
  4. 流式调用 LLM：LLM Provider 以流式方式调用底层大模型，实时产出 thinking（推理过程）、text（响应文本）和 tool_calls（工具调用）三类事件。
  5. 执行工具调用：若 LLM 产出 tool_calls，ToolRegistry 并发执行对应工具，收集执行结果。
  6. 持久化消息：MessageStore 将本轮 assistant 消息（含 thinking、text、tool_use、tool_result）通过 MongoDB $push 实时追加，防止进程崩溃导致数据丢失。
  7. 判断是否继续：若本轮存在 tool_calls，将工具结果追加到上下文后继续下一轮循环；若 LLM 输出纯文本回复，则循环结束，返回结果给用户。

ASCII 流程图

用户输入
  ↓
[SessionManager] 加载会话历史 (MongoDB)
  ↓
[PromptBuilder] 构建 system prompt + 消息历史
  ↓
[LLM Provider] 流式调用 LLM
  ├─→ thinking（推理过程）
  ├─→ text（响应文本）
  └─→ tool_calls（工具调用）
  ↓
[ToolRegistry] 执行工具
  ↓
[MessageStore] 持久化消息 ($push)
  ↓
如有 tool_calls → 继续循环，否则结束

3. LLM Provider 抽象层

为屏蔽不同大模型服务商的 API 差异，发仔定义了统一的 LLMProvider 抽象接口。 当前支持两种实现：

  • OpenAI：兼容 OpenAI Chat Completions API，支持 GPT 系列及兼容接口。
  • VolcEngine Ark：字节跳动火山引擎 Ark 平台，支持豆包等国内大模型。

所有 Provider 均以流式（Server-Sent Events）方式返回结果，并统一转换为以下标准流事件类型：

事件类型 说明 触发时机
THINKING_DELTA 推理过程增量片段 LLM 输出 thinking 内容时逐块触发
THINKING_DONE 推理过程完成 thinking 内容全部输出完毕
TEXT_DELTA 响应文本增量片段 LLM 输出正文内容时逐块触发
TEXT_DONE 响应文本完成 正文内容全部输出完毕
TOOL_CALL_PENDING 工具调用待执行 LLM 决定调用工具、参数组装完成后
TOOL_CALLS 工具调用列表完整 本轮所有工具调用参数均已就绪

4. SessionManager 与 Token 管理

会话历史加载

SessionManager 负责管理每个会话的完整消息历史。 消息持久存储于 MongoDB，同时以 Redis 作为热缓存层，加速高频访问场景下的历史读取。 每次 Agent Loop 启动时，SessionManager 优先从 Redis 读取缓存；若缓存未命中，则回退至 MongoDB 全量加载。

TokenTracker

TokenTracker 在每轮循环中追踪输入和输出的 token 用量，为 Context Compaction 提供决策依据， 同时用于计费统计与用量监控。

Context Compaction（上下文压缩）

当历史消息的 token 总量接近模型上下文窗口限制时，SessionManager 会自动触发上下文压缩策略：

  • 压缩阈值（COMPACT_THRESHOLD）为 0.85，即 token 用量超过上下文窗口的 85% 时触发。
  • 压缩时保留最近 10 条消息（COMPACT_KEEP_RECENT），对更早的历史进行摘要合并。
  • 至少需要 5 条消息（COMPACT_MIN_MESSAGES）才会触发压缩，避免对短会话无意义压缩。

图像剪裁

历史消息中可能包含图片（如截图、财务报表扫描件等），图片会消耗大量 token。 SessionManager 会自动识别并从较早的历史消息中移除图片内容， 仅保留最近消息中的图片，以有效节省 token 用量，同时不影响当前对话的图片理解能力。

5. 消息持久化机制

为保证消息不因进程崩溃或异常中断而丢失，发仔采用逐步 $push 策略进行消息持久化：

  • 每当 assistant 产出一段内容（thinking、text、tool_use 或 tool_result），立即通过 MongoDB 的 $push 操作追加到对应会话文档，而非等到整轮循环结束后批量写入。
  • 一条完整的 assistant 消息包含完整执行链：thinking → text → tool_use → tool_result，完整记录推理过程与工具调用细节，便于后续审计和调试。
  • 即便进程在某步骤中途崩溃，已 $push 的部分也已安全落盘，最大程度保护数据完整性。

设计原则：每一步 $push 而非最终批量写入，是发仔消息持久化的核心设计， 确保在高并发与不稳定网络环境下仍能保留尽可能完整的执行记录。

6. AgentLoop 关键配置

以下为 AgentLoop 上下文管理相关的核心配置项：

配置项 默认值 说明
COMPACT_CONTEXT_LIMIT 128000 token 模型上下文窗口总大小，压缩决策基于此值计算
COMPACT_OUTPUT_RESERVE 4096 token 为模型输出预留的 token 空间，不计入历史可用额度
COMPACT_THRESHOLD 0.85 触发上下文压缩的 token 占用比例阈值（85%）
COMPACT_KEEP_RECENT 10 条 压缩时保留的最近消息条数，确保近期上下文完整
COMPACT_MIN_MESSAGES 5 条 触发压缩所需的最少消息条数，避免短会话被误压缩
