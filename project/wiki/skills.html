Skills 插件系统

Skills 是发仔的可复用分析能力包，以 ZIP 压缩包形式分发， 可在运行时动态加载到沙箱执行环境中。每个 Skill 封装了一组领域专属的分析逻辑—— 例如股票技术指标计算、财务报表解析或 PDF 报告提取——Agent 通过 SkillTool 按需调用，无需将所有能力打包到同一个镜像中。

Skills 概述

Skills 系统的核心设计目标是让能力扩展与核心 Agent 解耦：新增、更新或废弃某项分析能力， 只需上传新版 Skill 包并在工作区配置中切换开关，无需重新部署 Agent 服务。

两种类型

类型 标识值 说明
内置 Skill builtin 由发仔团队统一维护，存放于代码仓库的 builtin_skills/ 目录， 通过 scripts/sync_builtin_skills.py 同步到数据库。 owner_id 为系统保留值 system。
用户自定义 Skill custom 由用户通过 API 上传，owner_id 为创建者用户 ID。 可灵活扩展私有或团队专属的分析能力，与内置 Skill 在同一调用链路中工作。

两种作用域

作用域 标识值 说明
全局 public 所有用户均可见并启用，通常用于内置 Skill 或团队共享的通用能力包。
工作区 private / team private 仅创建者可见；team 限定在指定团队（scope_id）内可见。 用于保护专有分析逻辑或限制能力仅在特定业务线使用。

工作区启用粒度：Skill 的 scope 控制谁能看到该 Skill， 而 agent_workspaces.skills 字段控制当前工作区是否启用该 Skill。 两者独立配置，互不干扰。

Skill 包结构

每个 Skill 以 ZIP 压缩包形式交付，内部文件布局有固定约定， manifest.yaml 是唯一的必填文件，其余文件按需包含： 

skill_name.zip
  manifest.yaml       # （必填）Skill 描述和工具定义
  main.py             # 主入口，实现 manifest 中声明的工具函数
  requirements.txt    # Python 依赖，沙箱加载时自动安装
  utils/              # 可选的辅助模块目录
    helpers.py
    constants.py
  data/               # 可选的静态数据文件（如行业分类映射表）
    industry_map.csv

manifest.yaml 关键字段

manifest.yaml 描述了 Skill 的元数据以及它向 Agent 暴露的工具接口， 是 SkillTool 解析和注册工具的唯一依据： 

name: skill_stock_analysis          # Skill 唯一标识（与数据库 _id 对应）
display_name: 股票技术分析            # 前端展示名称
version: 1.2.0                      # SemVer 版本号
description: |                      # 注入 LLM System Prompt 的功能描述
  提供 A 股、港股技术指标计算（MA、MACD、RSI 等）
  及 K 线形态识别能力，支持多周期分析。

tools:                              # 向 Agent 暴露的工具列表
  - name: calc_technical_indicators
    description: 计算指定股票的技术指标
    entry: main.calc_technical_indicators   # 模块路径.函数名
    parameters:
      type: object
      properties:
        symbol:
          type: string
          description: 股票代码，如 "600036.SH"
        indicators:
          type: array
          items:
            type: string
          description: 指标列表，如 ["MA5", "MACD", "RSI14"]
        period:
          type: string
          enum: [daily, weekly, monthly]
          description: K 线周期
      required: [symbol, indicators]

  - name: detect_candlestick_pattern
    description: 识别 K 线形态（如头肩顶、双底等）
    entry: main.detect_candlestick_pattern
    parameters:
      type: object
      properties:
        symbol:
          type: string
        lookback_days:
          type: integer
          description: 回溯交易日数量，默认 60
      required: [symbol]

entry 字段约定：格式为 module.function， 其中 module 是相对于 ZIP 根目录的 Python 模块路径（点分隔）， function 是该模块中的可调用对象名称。 例如 utils.helpers.parse_report 指向 utils/helpers.py 中的 parse_report 函数。

Skills 生命周期

一个 Skill 从创建到被 Agent 实际调用，经历以下六个阶段：

  1. 创建 / 上传 Skill 包（POST /api/skills）
    调用方将 Skill ZIP 包通过 multipart/form-data 上传到后端。 后端解析 manifest.yaml，验证字段合法性后将包存入 TOS （路径：skills/{name}/{version}.zip）， 并在 agent_skills 集合中写入元数据记录。 首次上传时 cur_version 自动设为该版本，后续上传新版本需手动晋升。
  2. 版本管理（版本号 + changelog）
    每次上传新版本时，旧版本记录保留在 versions[] 数组中，不会被覆盖。 每个 VersionEntry 包含版本号（SemVer）、TOS 包路径、发布时间以及 changelog 变更说明。运维人员可通过管理 API 将 cur_version 回滚到任意历史版本，以快速响应线上问题。
  3. 启用 / 禁用（PUT /api/skills/{id}/enable 或 disable）
    Skill 本身有全局 status（active / deprecated / draft）； 在工作区粒度，通过上述接口写入 agent_workspaces.skills[skill_id].enabled 字段， 控制该 Skill 是否对当前工作区的 Agent 可见。deprecated 状态的 Skill 无法被新工作区启用。
  4. 工作区配置（agent_workspaces.skills 字段）
    工作区文档的 skills 字段是一个 Map，记录每个 Skill 在该工作区中的配置： enabled（是否启用）与 loaded_version（当前已加载版本）。 新会话启动时，Agent 初始化逻辑读取此 Map，决定向 LLM 注入哪些工具描述。
  5. 沙箱加载（loaded_version 比对，按需同步）
    SkillTool 在首次被调用前，会比对工作区记录的 loaded_version 与 agent_skills.cur_version：若版本不一致，则从 TOS 拉取最新 ZIP 包， 解压到沙箱的 /workspace/skills/{skill_name}/ 目录， 并执行 pip install -r requirements.txt 安装依赖后，更新 loaded_version。相同版本下重复调用不触发重复下载，避免浪费 I/O。
  6. Agent 通过 SkillTool 调用
    Skill 加载完成后，SkillTool 根据 manifest.yaml 中的 entry 字段动态 import 对应模块，以关键字参数的方式调用目标函数， 将返回值序列化为字符串后作为工具结果返回给 LLM。整个调用过程在沙箱内完成， 与宿主环境完全隔离。

SkillDoc 数据模型

Skills 元数据存储于 MongoDB agent_skills 集合。每条文档的字段定义如下：

字段名 类型 说明
name String Skill 唯一标识（同时作为 _id），人类可读，建议使用蛇形命名， 如 skill_stock_analysis
description String Skill 功能描述，会被注入 LLM System Prompt， 帮助模型判断何时使用该 Skill 中的工具
cur_version String 当前稳定版本号（SemVer），新工作区配置该 Skill 时默认锁定此版本
type String 技能类型：builtin（内置，由系统维护）/ custom（用户自定义）
owner_id String 创建者用户 ID；内置 Skill 固定为系统保留值 system
scope String 可见范围：public（所有用户）/ private（仅创建者）/ team（指定团队，配合 scope_id 使用）
scope_id String 当 scope 为 team 时有效，记录目标团队 ID； 其他情况为空字符串
versions Array<VersionEntry> 历史版本列表，按发布时间升序排列；每个 VersionEntry 包含 version（版本号）、tos_path（ZIP 存储路径）、 released_at（发布时间）、changelog（变更说明）
status String 全局状态：active（可用）/ deprecated（已废弃，禁止新工作区启用）/ draft（草稿，仅 owner 可见和测试）

VersionEntry 子文档结构

{
  "version": "1.2.0",
  "tos_path": "tos://finance-agent-bucket/skills/skill_stock_analysis/1.2.0.zip",
  "released_at": "2024-03-15T08:30:00Z",
  "changelog": "新增 RSI 指标；修复 MACD 零轴判断逻辑错误"
}

内置 Skills 同步

内置 Skills 的源码存放于代码仓库，以保证版本追踪和 Code Review 流程可控。 scripts/sync_builtin_skills.py 脚本负责将仓库中的内置 Skill 同步到生产数据库，是 CI/CD 流水线的一部分，在每次合并到主干后自动触发。

脚本的主要工作

  1. 扫描内置 Skill 目录：遍历 builtin_skills/ 下的所有子目录， 读取每个 Skill 的 manifest.yaml，获取名称与版本。
  2. 比对数据库现有记录：查询 agent_skills 集合， 对比 name 和 cur_version。若版本已存在则跳过， 防止重复上传。
  3. 打包并上传到 TOS：对有变更的 Skill，脚本在临时目录中创建 ZIP 包， 按路径规范 skills/{name}/{version}.zip 上传到 TOS。
  4. 写入 / 更新数据库文档：向 agent_skills 的 versions[] 追加新版本条目，并将 cur_version 更新为最新版本， type 固定写为 builtin，scope 固定为 public。
  5. 输出同步报告：打印每个 Skill 的处理结果（新增 / 更新 / 跳过）， 供 CI 日志审计。

本地运行：开发环境中可手动执行 python scripts/sync_builtin_skills.py --dry-run 预览将要发生的变更， 加 --dry-run 参数不会实际写入数据库或上传 TOS。

如何开发新 Skill

以下步骤以开发一个"财务报表解析"Skill 为例，演示从本地编写到线上可用的完整流程。

步骤一：创建 manifest.yaml

在本地新建一个工作目录（如 skill_fin_report/）， 创建 manifest.yaml，定义工具名、参数与描述： 

name: skill_fin_report
display_name: 财务报表解析
version: 0.1.0
description: |
  解析 A 股上市公司财务报表（利润表、资产负债表、现金流量表），
  提取关键财务指标并计算同比环比变化，支持 PDF 和 Excel 两种格式输入。

tools:
  - name: parse_financial_report
    description: 解析财务报表文件，返回结构化财务数据
    entry: main.parse_financial_report
    parameters:
      type: object
      properties:
        file_path:
          type: string
          description: 报表文件在沙箱中的路径（相对于 /workspace/）
        report_type:
          type: string
          enum: [income_statement, balance_sheet, cash_flow]
          description: 报表类型
        fiscal_year:
          type: integer
          description: 财年，如 2023
      required: [file_path, report_type]

步骤二：编写实现代码

在同一目录下创建 main.py，实现 manifest.yaml 中声明的函数。 函数参数必须与 parameters 中定义的字段一一对应： 

import json
from pathlib import Path

def parse_financial_report(
    file_path: str,
    report_type: str,
    fiscal_year: int = None
) -> str:
    """
    解析财务报表，返回 JSON 字符串格式的结构化数据。
    SkillTool 会将返回值直接作为工具结果传给 LLM，
    建议返回人类可读的文本或紧凑 JSON。
    """
    abs_path = Path("/workspace") / file_path
    if not abs_path.exists():
        return f"错误：文件不存在 {file_path}"

    # 根据文件后缀选择解析器
    suffix = abs_path.suffix.lower()
    if suffix == ".pdf":
        data = _parse_pdf(abs_path, report_type)
    elif suffix in (".xlsx", ".xls"):
        data = _parse_excel(abs_path, report_type)
    else:
        return f"错误：不支持的文件格式 {suffix}"

    if fiscal_year:
        data = {k: v for k, v in data.items() if v.get("year") == fiscal_year}

    return json.dumps(data, ensure_ascii=False, indent=2)

步骤三：打包成 ZIP

进入 Skill 工作目录，将所有文件压缩为 ZIP 包。 注意：ZIP 包内的文件应直接位于根层，不要套一层同名目录， 否则 SkillTool 在解压后无法正确解析模块路径： 

# 正确：从 skill_fin_report/ 目录内部打包
cd skill_fin_report/
zip -r ../skill_fin_report_v0.1.0.zip .

# 验证包内容（应看到 manifest.yaml 在根层）
unzip -l ../skill_fin_report_v0.1.0.zip

步骤四：通过 API 上传

使用 POST /api/skills 接口将 ZIP 包上传到系统。 请求需携带有效的认证 Token，上传成功后接口返回新建的 SkillDoc： 

curl -X POST https://finance-agent.example.com/api/skills \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@skill_fin_report_v0.1.0.zip" \
  -F "scope=private"

响应示例：

{
  "name": "skill_fin_report",
  "display_name": "财务报表解析",
  "cur_version": "0.1.0",
  "type": "custom",
  "owner_id": "user_abc123",
  "scope": "private",
  "scope_id": "",
  "status": "active"
}

步骤五：在工作区启用

上传后 Skill 默认未在任何工作区启用，需要调用启用接口将其加入目标工作区： 

curl -X PUT https://finance-agent.example.com/api/skills/skill_fin_report/enable \
  -H "Authorization: Bearer $TOKEN" \
  -H "X-Workspace-Id: workspace_xyz789"

启用后，下次在该工作区发起会话时，Agent 将能感知到 parse_financial_report 工具，并在适当时机自动调用它。 首次调用时 SkillTool 会自动将 ZIP 包下载并解压到沙箱，后续重复调用直接执行，无需重新加载。

调试技巧：开发阶段可将 Skill 的 status 设为 draft，此时仅 owner_id 对应的用户可见和测试， 确认无误后再发布为 active 对团队或全局开放。
