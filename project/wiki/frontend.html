前端架构

Finance Agent（发仔）的前端基于 React 19 构建，采用现代化的工程化体系， 通过 Vite 实现极速开发体验，Zustand 管理全局状态，TipTap 支持富文本编辑， 整体架构遵循「功能域划分」原则，各功能域独立封装、按需加载。

技术栈

技术 版本 用途
React 19 UI 框架，利用并发渲染与 Server Components 特性提升响应速度
TypeScript 5.9 全量静态类型，strict 模式，确保类型安全与可维护性
Vite 7 构建工具，原生 ESM + esbuild 极速 HMR，生产环境 Rollup 打包
Tailwind CSS 4 原子化 CSS，JIT 引擎按需生成样式，支持暗色主题切换
Zustand 5 轻量全局状态管理，替代 Redux，配合 immer 实现不可变更新
react-router-dom 7 客户端路由，支持 lazy() 代码分割与嵌套路由
TipTap 2 富文本编辑器，基于 ProseMirror，用于消息输入与文档编辑
Vitest — 单元测试框架，与 Vite 共享配置，支持 jsdom 环境测试 React 组件

路由设计

路由基于 react-router-dom v7，所有一级页面均挂载在 /agent 前缀下，通过 lazy() + <Suspense> 实现路由级代码分割，确保首屏只加载当前路由所需的 JS Bundle。

路径 页面组件 说明
/agent AgentPage 主聊天界面，展示会话列表与消息流；未登录时重定向至登录页
/agent/workspace/:workspaceId WorkspacePage 工作区详情页，包含文件树、代码查看与 Agent 执行面板
/agent/admin/* AdminPage 管理后台，包含用户管理、Skills 配置、系统监控等子路由；需管理员权限
/agent/onboarding OnboardingPage 新用户引导流程，完成后写入 profileStore 并跳转至主页
/agent/share/:shareToken SharePage 会话分享只读页，无需登录即可访问，通过 shareToken 鉴权

代码分割策略

所有顶层路由组件均通过 React.lazy() 动态导入，外层用 <Suspense fallback={<Spinner />}> 包裹， Vite 会自动将每个 lazy() 边界拆分为独立 Chunk， 仅在用户首次导航到对应路由时按需加载： 

import { lazy, Suspense } from 'react';
import { createBrowserRouter, RouterProvider } from 'react-router-dom';
import Spinner from '@/components/Spinner';

const AgentPage      = lazy(() => import('@/pages/AgentPage'));
const WorkspacePage  = lazy(() => import('@/pages/WorkspacePage'));
const AdminPage      = lazy(() => import('@/pages/AdminPage'));
const OnboardingPage = lazy(() => import('@/pages/OnboardingPage'));
const SharePage      = lazy(() => import('@/pages/SharePage'));

const router = createBrowserRouter([
  {
    path: '/agent',
    element: <Suspense fallback={<Spinner />}><AgentPage /></Suspense>,
  },
  {
    path: '/agent/workspace/:workspaceId',
    element: <Suspense fallback={<Spinner />}><WorkspacePage /></Suspense>,
  },
  {
    path: '/agent/admin/*',
    element: <Suspense fallback={<Spinner />}><AdminPage /></Suspense>,
  },
  {
    path: '/agent/onboarding',
    element: <Suspense fallback={<Spinner />}><OnboardingPage /></Suspense>,
  },
  {
    path: '/agent/share/:shareToken',
    element: <Suspense fallback={<Spinner />}><SharePage /></Suspense>,
  },
]);

export default function App() {
  return <RouterProvider router={router} />;
}

全局 Store（Zustand）

全局状态由 6 个职责单一的 Zustand Store 管理，每个 Store 对应一个独立文件， Store 之间不直接互相订阅，通过组件层或 Hooks 层协调交互。 所有 Store 均启用 immer 中间件以支持不可变更新， 敏感状态（如会话 token）不持久化到 localStorage。

Store 文件 职责
chatStore chatStore.ts 消息列表、会话状态、SSE 连接管理；包含流式消息 reducer，处理 token / tool_call / done 等事件
workspaceStore workspaceStore.ts 工作区元数据、文件索引树、当前打开文件；负责文件上传与删除的状态同步
profileStore profileStore.ts 当前登录用户的 ID、昵称、头像、角色权限；登录 / 登出时清空
settingsStore settingsStore.ts 用户偏好配置，包含主题（light / dark / system）、语言、消息字体大小等；持久化至 localStorage
toastStore toastStore.ts Toast 通知队列；提供 toast.success()、toast.error()、toast.info() 快捷方法
traceStore traceStore.ts Agent 执行 Trace 面板数据；记录工具调用链路、耗时与中间结果，供调试面板展示

核心组件说明

StreamingText

StreamingText 是消息渲染的核心组件，负责将 Agent 返回的 Markdown 文本 实时解析并渲染为结构化 UI。组件支持以下 7 种内容类型的富展示：

  • 普通文本：支持 Markdown 标准语法（标题、加粗、斜体、列表等）
  • 代码块：语法高亮（基于 highlight.js），支持一键复制，可折叠长代码
  • 图片：内联展示，点击可触发 FilePreview 全屏预览
  • 文件链接：渲染为 InlineFileWidget 卡片，展示文件名、大小与下载按钮
  • 表格：响应式表格渲染，超宽时横向滚动，超长时虚拟滚动
  • 数学公式：通过 KaTeX 渲染 LaTeX 公式（$...$ 内联，$$...$$ 块级）
  • 工具调用卡片：展示 Agent 的 tool_call / tool_result 步骤，可展开查看原始参数与输出

流式渲染期间，组件通过增量 diff 策略只更新新增节点，避免全量重渲染导致的闪烁。

InlineFileWidget

文件附件卡片组件，展示单个文件的元数据信息：文件类型图标、文件名、文件大小。 卡片内置「预览」与「下载」两个操作按钮。点击预览时根据 MIME 类型分发至 FilePreview 组件；点击下载时通过签名 URL 触发浏览器下载， 下载过程中按钮展示 loading 状态（遵循操作类接口 loading 规范）。

FilePreview

文件预览模态框，根据文件类型选择对应的渲染策略：

  • 图片（PNG / JPG / SVG）：全屏展示，支持缩放与拖拽
  • PDF：基于 react-pdf 分页渲染，支持翻页与缩放
  • CSV / Excel：解析为表格数据后以虚拟滚动表格展示，支持列排序
  • 其他类型：展示文件元数据与下载入口，不做内容渲染

FileTree

工作区文件浏览器，以树形结构展示 workspaceStore 中的文件索引。 支持展开 / 折叠目录、单击选中文件（更新 workspaceStore.activeFileId）、 右键上下文菜单（重命名、删除、下载）。文件状态（上传中 / 处理中 / 就绪） 通过图标颜色区分，上传进度通过节点内 progress bar 实时展示。

SkillMarketplace

Skills 市场面板，展示当前工作区已启用及可安装的 Skills 列表。 每个 Skill 卡片展示名称、描述、版本与状态。 启用 / 禁用操作通过 withLoading() 包裹，操作期间卡片进入 loading 状态， 接口成功响应后才更新 UI（禁止乐观更新）。

CommandPalette

命令搜索面板，通过 ⌘K（macOS）/ Ctrl+K（Windows）快捷键唤起。 支持模糊搜索以下资源：历史会话、工作区文件、可用 Skills、导航页面。 搜索结果按类别分组展示，键盘上下方向键导航，回车执行选中项。 搜索逻辑运行在 Web Worker 中，避免阻塞主线程。

SSE 流式处理

前端通过 utils/sse.ts → chatStore.ts reducer 两层管道消费后端推送的 SSE 事件，实现流式消息的增量渲染。

utils/sse.ts — 连接管理

sse.ts 封装原生 EventSource API，提供带鉴权支持的 SSE 连接工厂函数。 由于原生 EventSource 不支持自定义请求头，此处改用 fetch + ReadableStream 模拟 SSE， 将 Authorization: Bearer <token> 注入请求头： 

// utils/sse.ts
export async function* createSSEStream(
  url: string,
  token: string,
  signal: AbortSignal,
): AsyncGenerator<SSEEvent> {
  const response = await fetch(url, {
    headers: { Authorization: `Bearer ${token}` },
    signal,
  });
  const reader = response.body!.getReader();
  const decoder = new TextDecoder();
  let buffer = '';

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    buffer += decoder.decode(value, { stream: true });
    const lines = buffer.split('\n\n');
    buffer = lines.pop() ?? '';
    for (const chunk of lines) {
      const dataLine = chunk.split('\n').find(l => l.startsWith('data:'));
      if (dataLine) yield JSON.parse(dataLine.slice(5)) as SSEEvent;
    }
  }
}

chatStore.ts — reducer 模式状态管理

chatStore 内置 handleSSEEvent() reducer， 按事件类型分发消息状态更新，保证消息列表的不可变更新与最小重渲染范围： 

// stores/chatStore.ts（片段）
function handleSSEEvent(state: ChatState, event: SSEEvent): ChatState {
  switch (event.type) {
    case 'token':
      // 追加 token 到当前流式消息末尾
      return appendTokenToLastMessage(state, event.data);
    case 'tool_call':
      // 插入工具调用卡片（pending 状态）
      return appendToolCallMessage(state, event.data);
    case 'tool_result':
      // 更新对应工具调用卡片为 done 状态
      return resolveToolCallMessage(state, event.data);
    case 'error':
      // 标记当前消息为 error 状态，展示错误提示
      return markMessageError(state, event.data);
    case 'done':
      // 流结束，将流式消息转为静态消息，关闭 SSE 连接
      return finalizeStreamingMessage(state);
    default:
      return state;
  }
}

startChat() action 负责创建 SSE 流并驱动 reducer 循环： 

// stores/chatStore.ts（片段）
startChat: async (sessionId: string, content: string) => {
  const abortController = new AbortController();
  set({ abortController, isStreaming: true });

  const token = profileStore.getState().accessToken;
  const stream = createSSEStream(`/api/chat/${sessionId}`, token, abortController.signal);

  for await (const event of stream) {
    set(state => handleSSEEvent(state, event));
    if (event.type === 'done') break;
  }

  set({ isStreaming: false, abortController: null });

开发规范要点

操作类接口必须有 loading 状态

所有触发后端写操作（创建、更新、删除）的函数， 必须在调用前将对应状态置为 loading: true， 接口返回（无论成功或失败）后恢复为 loading: false。 组件侧根据 loading 状态禁用按钮或展示 Spinner，防止用户重复提交。

使用 withLoading() 辅助函数

storeHelpers.ts 提供 withLoading() 高阶函数， 自动管理 loading 状态的开启与关闭，并统一捕获异常后通过 toastStore 展示错误提示： 

// stores/storeHelpers.ts
export function withLoading<T>(
  set: (partial: Partial<{ loading: boolean }>) => void,
  fn: () => Promise<T>,
): Promise<T | undefined> {
  set({ loading: true });
  return fn()
    .catch((err: unknown) => {
      const message = err instanceof Error ? err.message : '操作失败，请重试';
      toastStore.getState().toast.error(message);
      return undefined;
    })
    .finally(() => {
      set({ loading: false });
    });
}

// 使用示例（workspaceStore.ts）
deleteFile: (fileId: string) =>
  withLoading(set, async () => {
    await api.deleteFile(fileId);
    set(state => {
      state.files = state.files.filter(f => f.id !== fileId);
    });
  }),

Toast 使用规范

所有面向用户的操作反馈（成功提示、错误提示、信息通知） 均通过 toastStore 中的 toast 对象触发， 禁止直接操作 DOM 或使用第三方 toast 库： 

import { toastStore } from '@/stores/toastStore';

const { toast } = toastStore.getState();

toast.success('文件上传成功');
toast.error('网络异常，请稍后重试');
toast.info('正在处理中，请等待...');

禁止乐观更新

发仔前端严格禁止乐观更新（Optimistic Update）——即在接口返回成功之前提前修改 UI 状态。 原因是 Agent 的操作涉及文件处理、沙箱执行等异步副作用， 乐观更新在回滚时会导致与后端状态不一致，增加调试难度。

规范：接口调用期间展示 loading 状态； 接口返回 code: 0（成功）后，用服务端返回的数据更新 Store； 接口失败时通过 toast.error() 提示用户，Store 状态保持不变。
